# GitHub MCP + `plan.md` as the single source the dashboard reads

**Date:** 2026-06-29
**Branch:** `replan/tp60-palencia-2026` → target `main`
**Repo:** `Auriga13/plantrail` (GitHub Pages deploys from `main`)

## 1. Purpose

Let Claude on **claude.ai web** read and **modify the training plan that lives in GitHub**,
and make that same `plan.md` the **single source** the hosted dashboard renders from — so an
edit made from the web is the same file shown on the site. No duplicate plan, no drift.

Two halves:
- **Connection:** add GitHub's official remote MCP server as a custom connector in claude.ai
  so web-Claude can read/commit `plan.md` in the repo.
- **Coherence:** retire the hardcoded Python `PLAN` and have the dashboard build read the
  plan out of `plan.md`, so edits become visible on the next deploy.

## 2. Context (verified)

- `plan.md` exists only locally on the unpushed `replan/tp60-palencia-2026` branch — **not on
  GitHub yet.** It must reach `main` for any GitHub-side tooling to see it.
- The dashboard is generated by `trail_analyzer.py` and `plan_completo.py`, each carrying its
  own hardcoded `PLAN` list (`trail_analyzer.py:201`, `plan_completo.py:43`) + `PLAN_START`
  (`:67` / `:20`), baked into `dashboard.html` / `plan_completo.html`.
- `PLAN` week schema: `{week, phase, phase_name, load, title, km, d_plus, sessions:[{day,
  type, km, d, desc}], notes}`. Dashboard JS also consumes a reduced
  `[{week, phase, km, d_plus}]` JSON for plan-vs-actual charts.
- `.github/workflows/daily-update.yml` runs `ci_refresh.py` then `plan_completo.py` +
  `trail_analyzer.py` on a cron (and manual dispatch), committing regenerated artifacts.
- GitHub's MCP server is free for all GitHub users (no Copilot needed); tools inherit normal
  repo permissions. Security is low-stakes here ("just a plan").

## 3. Decisions (from brainstorming)

- **Single source = `plan.md`**, containing human prose **plus one fenced ` ```yaml ` block**
  holding the full 15-week structured plan (same schema as today's `PLAN`). The YAML block is
  machine-read; the prose is the human "why." Brittle prose-parsing is avoided.
- **Drop the redundant hand-written weekly tables** from the markdown prose; the structured
  weeks live in the YAML block and the dashboard is their visual rendering. (Goals,
  philosophy, periodization summary, adjustment log stay as prose.)
- **Full coherence:** the Python files import `PLAN` from the new loader; the hardcoded lists
  are deleted.
- **Connection path:** try the GitHub **remote** MCP connector in claude.ai first; if its
  OAuth handshake won't complete, fall back to the local GitHub MCP server with a
  repo-scoped PAT (Claude Desktop / this CLI).
- **Edit target branch = `main`** (what Pages deploys and what "the plan in GitHub" means).

## 4. Architecture

### 4.1 `plan.md` structure (single source)
- Prose: §A Goals & constraints, §B Philosophy (the modern core), §C Periodization summary,
  §E Adjustment log.
- **§ Plan data — one fenced ` ```yaml ` block** with:
  - `plan_start: 2026-06-29`
  - `weeks:` — 15 entries, each `{week, phase, phase_name, load, title, km, d_plus,
    sessions: [{day, type, km, d, desc}], notes}`.
- The block is the authoritative weekly data. The current-week prose table is removed (the
  dashboard renders the current week from the block).

### 4.2 `plan_loader.py` (new, small, single-purpose)
- `load_plan(path="plan.md") -> (plan_start: date, plan: list[dict])`.
- Extract the first ` ```yaml ` fenced block from `plan.md`, parse with `yaml.safe_load`,
  validate shape, return `PLAN_START` and `PLAN` in the exact structure the renderers expect.
- Raises a clear error if the block is missing, malformed, or fails validation (week count,
  required keys, types). No silent fallback to a stale plan.
- Dependency: `pyyaml` (add to deps / `.env.example`/requirements as used by the workflow).

### 4.3 Renderer refactor
- `trail_analyzer.py` and `plan_completo.py`: replace the hardcoded `PLAN`/`PLAN_START` with
  `PLAN_START, PLAN = load_plan()`. No change to rendering logic or the reduced JSON.
- Keep all Strava-analysis logic in Python (only the *plan* moves to `plan.md`).

### 4.4 Connection (GitHub MCP)
- Add GitHub remote MCP server as a custom connector in claude.ai (Max). Authorize against
  `Auriga13/plantrail`.
- Verify Claude can `get` `plan.md` and `create-or-update file` (commit) on `main`.
- Fallback documented: local GitHub MCP server + repo-scoped PAT if remote OAuth fails.

### 4.5 Web re-plan workflow (end state)
From claude.ai web: user asks "review last week / set W#" → Claude reads `plan.md` (GitHub
MCP) + Strava (Strava MCP) → computes signals (ACWR, volume/D+, time-in-zone, decoupling) →
proposes → on approval **commits edited `plan.md` to `main`** (updates the YAML week + appends
to §E) → daily workflow (or manual dispatch) regenerates the dashboard from `plan.md`.

## 5. Migration / sequencing

1. Convert the existing `PLAN` (15 weeks) into the YAML block inside `plan.md`; remove the
   redundant prose weekly tables.
2. Add `plan_loader.py` + `pyyaml`.
3. Refactor both renderers to use the loader; **delete** hardcoded `PLAN`/`PLAN_START`.
4. Update tests (`tests/test_plan.py`, `tests/test_plan_completo.py`) to assert the loader
   yields 15 weeks numbered 1–15 and `plan_start == 2026-06-29` (parity with old behavior).
5. Regenerate dashboards locally; confirm byte-for-byte-equivalent plan content vs current.
6. Land `plan.md` + code on `main` (merge the branch).
7. Add the GitHub MCP connector in claude.ai; test read + commit on `main`.

## 6. Out of scope

- Auto-running the deploy on every MCP commit (rely on existing daily cron + manual dispatch;
  optionally a later `on: push` trigger for `plan.md`).
- Nutrition as a separate file/tool (the plan covers nutrition inline for now; a dedicated
  nutrition source can follow the same pattern later).
- Any in-page (browser-side) chat — the connector model replaces that idea.

## 7. Risks

- **Remote GitHub MCP OAuth may not connect from claude.ai** → fallback to local PAT server
  (§4.4). Test early.
- **Parser fragility** → mitigated by using an exact YAML block (not prose) + validation +
  tests.
- **Plan parity regression** → mitigated by regenerating and diffing the dashboards in step 5.

## 8. Success criteria

- `plan.md` (prose + YAML block) is on `main`; both renderers read the plan from it; the
  hardcoded `PLAN` lists are gone; tests pass.
- Regenerated `dashboard.html` / `plan_completo.html` show the same plan as before the refactor.
- From claude.ai web, Claude reads `plan.md` and commits an edit to `main` via the GitHub MCP;
  after a deploy, the dashboard reflects the change.
- Editing the plan anywhere means editing exactly one file in GitHub — `plan.md`.
